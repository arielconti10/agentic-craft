## Staleness risks

- **No product anchors cuts both ways**: nothing can go stale by name, but the guide silently encodes a March–June 2026 snapshot of one interaction idiom (single-thread, in-conversation agent) with no way for a reader to date it.
- **Plans**: plan/todo UI is the fastest-moving surface (Claude Code plan mode, Cursor agent todos, ChatGPT agent mode all shipped or reworked plan UIs across 2025–26). The "editable plan card with drag handles" prescription is demo-fiction — the demo doesn't even implement drag — and may already diverge from how current products handle plan revision (chat-driven regeneration, not direct manipulation).
- **Parallel/subagents**: the tree-connector model assumes parallelism happens *inside one thread*. The industry has moved substantially toward parallel *sessions* (multiple concurrent agents in tabs/grids/worktrees — Cursor background agents, multi-Claude-Code, Devin fleets). The section covers the small case and misses the dominant one; the separate `multi-agent-content.tsx` view may cover it, but this section doesn't cross-reference it.
- **Tool calls**: "never a function signature" predates/ignores the trend toward *showing raw commands for trust* in developer agents; as agentic IDEs standardize on literal-command display with approve-to-run, the absolutist rule reads increasingly consumer-only.
- **Approval gates**: "Cost / time: $0.09 estimated" bakes in token-pricing assumptions that shift quarterly; also, permission systems have been moving from per-action modals toward policy/autonomy settings (allowlists, auto-accept tiers) — the guide has an `autonomy-contract` view elsewhere but this section presents the modal as the pattern.
- Demo data is dated "2026-03" throughout — cosmetic, but signals age if published unrevised.

## Quality read

This is better than pattern-catalog filler on the build side — every section is a working, stateful demo, and four of seven patterns are backed by genuinely reusable compound components — but as *reference content* it is entirely taste-based: not one claim names a product, cites a source, or distinguishes observation from invention, and a couple of spec rules are contradicted by the guide's own code (monochrome icons vs. destructive-red failures; "N of M, not percentage" next to a percentage progress bar). The single strongest idea is the approval gate's payload semantics — the "locked payload" whose approval *expires if source material changes*, plus consequence/reversibility/rollback as first-class preview fields — which is a defensible, near-novel contribution; the consequence-annotated decision options are a close second. The weakest sections (tool calls, plans, parallel) describe what every 2025-era agent UI already does and would need product-grounded specifics or a stated point of view to be worth publishing.

## CONFIRMED

**1. Plans are editable (add/remove/reorder) before execution — real cross-product pattern.**
- Cursor — Plan Mode: "an interactive editor to modify plans inline. ## CONTRADICTED / STALE

**1. "Tool labels plain-language, never raw function signatures" — contradicted as an absolute; audience-dependent.**
Developer agents deliberately show the literal payload: Claude Code shows the exact bash command and the exact diff before running (https://code.claude.com/docs/en/permissions); Codex approval flow surfaces the specific command/edit for approval, with `/approve` operating on recorded denied *commands* (https://developers.openai.com/codex/agent-approvals-security, https://developers.openai.com/codex/cli/slash-commands); Cursor's agent shows terminal commands it wants to run. The trend in agentic IDEs is literal-command display *for trust*. The rule survives only if scoped: "plain-language label as the collapsed header, raw payload one click away — and for consequential developer actions, show the literal command."

**2. "All-monochrome icons, no colored status indicators" — contradicted, including by the guide's own code.**
Claude Code's own task display uses colored status markers (✅ completed / 🔧 in-progress / ❌ pending per third-party documentation: https://claudefa.st/blog/guide/development/todo-workflows) and green/red diffs; Cursor renders green/red added/removed line stats on every agent edit (visible in every changelog screenshot: https://cursor.com/changelog/2-0); Linear's agent system exposes colored rich statuses (working / waiting for input / error / done): https://linear.app/developers/agent-interaction. No panel product observed enforcing monochrome status. The guide also colors failed calls `text-destructive`, contradicting itself. Defensible only as a stated *taste position* ("reserve color for failure"), not as observed practice.

**3. Subagent progress "N of M unit, not percentage" + nested tool calls inline — contradicted by the flagship subagent product.**
Claude Code's shipped subagent row shows **elapsed time + token count**, not N-of-M: "Running agent CLAUDE.md compliance review · 2m 34s · ↓ 2.2k tokens", with no per-subagent task breakdown; users are asking for more (open feature request, 2026: https://github.com/anthropics/claude-code/issues/48246). The SDK docs state intermediate tool calls **stay inside the subagent** — only the final message returns to the parent thread (https://code.claude.com/docs/en/agent-sdk/subagents). So both the "own concrete progress indicator" and "nested tool calls render inline" prescriptions describe something richer than what the reference products ship. (eve likewise emits `subagent.called`/`subagent.completed` with a `childSessionId` to *attach to* — a separate session, not inline nesting: `node_modules/eve/docs/concepts/sessions-runs-and-streaming.md`.)

**4. "Never auto-select — always wait for explicit user choice" — contradicted as universal.**
Devin's plan gate **auto-proceeds after 30 seconds by default** ("By default, Devin will wait thirty seconds for feedback from you before automatically proceeding"; opt-in "Wait for my approval" for a hard gate; configurable in Settings → Customization): https://docs.devin.ai/work-with-devin/interactive-planning. Claude Code's AskUserQuestion has a 60s timeout and marks a "(Recommended)" default (https://smartscope.blog/en/generative-ai/claude/claude-code-askuserquestion-tool-guide/). The real shipped spectrum is hard gate ↔ soft gate with timeout ↔ recommended-default — the guide only covers the hard gate. (The "Impact:" line per option remains the guide's invention — no product found annotating per-option consequences; that part is a defensible proposal if labeled as such.)

**5. Plan card details — partially stale.**
- Drag handles for reordering: no product found. Cursor's plan editing is inline text editing of a plan that is literally a file on disk ("Agent plans are now files that can be edited with normal tools" — https://cursor.com/changelog); Claude Code is chat-feedback + markdown; Devin is feedback/suggestion-driven. The demo's fake drag handle would publish as fiction.
- "6px square dot, not a checkbox or number": Claude Code uses checkbox-style glyphs/emoji; Devin and Replit use checkmarks; no square-dot precedent found. Pure taste — fine if stated as such.
- Staleness confirmed: Claude Code replaced its Todo system with **Tasks** (dependency tracking, filesystem persistence, cross-session) in v2.1.16, Jan 22, 2026 (https://claudefa.st/blog/guide/development/todo-workflows) — this surface churns quarterly.

**6. Parallel execution = in-thread tree with connectors — stale framing; the dominant shipped pattern is parallel *sessions*.**
Cursor 2.0 (Oct 2025): up to 8 parallel agents per prompt via git worktrees/remote machines (https://cursor.com/blog/2-0). Cursor 3.0 (April 2, 2026): the **Agents Window** — "run many agents in parallel across repos and environments: locally, in worktrees, in the cloud, and on remote SSH", with agent tabs side-by-side or in a grid (https://cursor.com/changelog/3-0). Cursor 04-24-26 changelog: async `/multitask` spawning subagents, redesigned worktrees, multi-root workspaces (https://cursor.com/changelog/04-24-26). Manus Wide Research (July 2025): 100+ parallel general-purpose subagents, each in its own VM (https://manus.im/blog/introducing-wide-research). The guide's in-thread tree covers the small case (parallel tool calls within one turn) and omits the industry-dominant one entirely; the section must at minimum cross-reference the multi-session model.

**7. Approval preview taxonomy (Consequence / Reversibility / Rollback / Cost-time / Permission / Source as first-class fields) — no shipped product does this.**
Claude Code shows command + diff; Codex shows the command and a sandbox/approval-mode context; ChatGPT agent asks in natural language and confirms purchase details (order/shipping/payment — the one shipped structured confirm: https://openai.com/index/buy-it-in-chatgpt/). Nobody ships rollback plans or cost estimates in approval previews. This is the guide's strongest *invention* — publish it explicitly as a proposal, and drop or generalize the "$0.09 estimated" token-pricing framing (stale within a quarter).

**8. Per-action modal as *the* approval pattern — stale relative to the policy-tier direction.**
Claude Code: allow/ask/deny permission *rules* evaluated deny→ask→allow, "approve and remember" writes rules to settings (https://code.claude.com/docs/en/permissions, /docs/en/agent-sdk/user-input). Codex: approval presets (Read Only / Auto / Full Access) plus an auto-review mode where **a reviewer agent replaces the human at the approval boundary** (https://developers.openai.com/codex/agent-approvals-security). eve: per-tool `needsApproval` policy including input-dependent predicates (`({toolInput}) => toolInput.amount > 1000`) and `once()` (first-call-only approval). The modal is one tier in a policy system, not the pattern.
