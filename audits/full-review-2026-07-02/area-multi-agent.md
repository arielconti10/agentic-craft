## Patterns asserted

1. **Agent Cards** (label: Identity, `#agent-cards`)
   - Core claim: Each agent needs an identity surface showing name, role, current status, and active task, "reflecting the agent's operational state in real time."
   - Prescription: Render per-agent cards with three exclusive states — active (task + progress count), idle (a *reason* for idleness, e.g. "waiting for a source packet to map"), error (failure message + inline Retry button). Pair with an operational table adding Status / Task / Progress% / Cost($) / Updated columns.

2. **Handoff Flow** (label: Orchestration, `#handoff-flow`)
   - Core claim: Sequential handoff means each step's output becomes the next agent's input, and the handoff itself should be reified as a first-class artifact ("packet") with an ownership-transfer lifecycle.
   - Prescription: Show a `HandoffPacket` with sender/receiver identity, status (draft → sent → accepted/rejected/expired), and structured fields: Payload, Source basis, Receiver action, Recovery (what rejection does). Show Accept/Reject buttons only in the `sent` state. Below it, a step pipeline with per-step state (pending/active/complete) and agent attribution.

3. **Parallel Agents** (label: Concurrency, `#parallel-agents`)
   - Core claim: Concurrently executing agents each report individual progress toward independent objectives.
   - Prescription: Per-agent row with pulse dot while running, determinate percent progress bar, and on completion a tick plus a *result line that replaces the task line* ("847 incidents scanned — 0 critical blockers").

4. **Agent Routing** (label: Dispatch, `#agent-routing`)
   - Core claim: Incoming tasks are routed to the most appropriate agent by task-type analysis, or manually by the workflow owner.
   - Prescription: Show the incoming task with its detected task type; in auto mode highlight the chosen agent with a one-line rationale ("Best match — requirement expertise") and dim non-candidates to 40% opacity; in manual mode make all agents selectable.

5. **Agent Communication** (label: Collaboration, `#agent-communication`)
   - Core claim: Inter-agent exchange happens via direct messaging or a shared context workspace, and "both patterns create auditable communication records."
   - Prescription: Direct mode — a chat-style thread with explicit from→to labeling per message. Shared mode — a typed artifact feed (Artifact / Finding / Draft / Analysis / Assessment badges) with agent attribution and relative timestamps.

6. **Workflow Runs** (label: Orchestration, `#workflow-runs`) — by far the densest section; a 10-row "Contract" spec table:
   - Core claim: A script-orchestrated run should be monitored as declared phases fanning out to parallel agents, answering "is this run healthy, where is it, what is it costing?"
   - Prescriptions (each a named contract): (a) fixed phase-status vocabulary (tick/pulse/dashed/alert) with `aria-current="step"` on at most one phase and a per-agent dot minimap; (b) phases are display groups, not barriers — barriers come from `parallel()`, so two phases can be active at once; (c) the orchestration script never renders ambiently, staying one disclosure away as provenance (Plan view with gutter glyphs and a live-phase band); (d) ≤5 agent rows per phase, 6+ collapse into a +N summary that carries its own status/token roll-up; (e) inline agent detail disclosure — schema-validated returns in mono, prose in serif, runtime semantics in a muted system voice "never as the agent"; (f) roll-up math: phase tokens = Σ agent tokens, run = Σ phases; retried phases restart their roll-up, failed-attempt spend lives only on the run meter; (g) the journal caches *completed* agent calls only — resume replays them labelled "cached," in-flight agents re-run whole; (h) Pause reachable in ≤1 click while live, with an honest verb (nothing destroyed); (i) agents self-retry 3× before a phase fails; Retry re-runs only unfinished agents, Skip preserves the failed record, both "price themselves in the banner before commitment"; (j) ambient motion reserved for live state, frozen on pause, disabled under `prefers-reduced-motion`, everything else 150–200 ms.

## Staleness risks

- **The section presents multi-agent UI as speculative pattern-space, but by mid-2026 it's shipped reality** (Claude Code subagents/agent teams, Cursor's parallel agents, Copilot coding agent, Devin's multi-session runs). Any published version must reconcile with — or explicitly critique — what those products actually converged on, or it reads as pre-2025.
- **"Handoff" is now a term of art** (OpenAI Agents SDK: silent, non-negotiated control transfer). The guide's accept/reject-packet meaning collides with the established meaning; needs a naming or framing note.
- **No mention of A2A/MCP.** An inter-agent-communication section written in 2026 that doesn't position itself relative to the A2A task lifecycle (which has real accept/reject-adjacent states) will look under-researched.
- **Determinate progress bars and dollar costs per agent** both cut against where products landed (indeterminate activity indicators; token/credit/ACU accounting). The `$0.14` / `67%` demo values are the most likely to date the material.
- **Identical robot icon for every agent** undercuts the section's own "Identity" thesis; real products differentiate agents by avatar/color/model badge.
- Demo copy references "Incident-2025-3891" and "Launch Policy v1.2/v2" inconsistently across sections (v1.2 in Communication, v2 in Handoff) — minor, but an auditor-bait inconsistency.

## Quality read

This section is two things stapled together: four generic pattern-catalog entries (cards, parallel, routing, communication — thin prose, hardcoded demos, no reusable components, nothing a designer couldn't invent in an afternoon) and two genuinely distinctive contributions backed by serious code — the HandoffPacket as a negotiated, recoverable ownership-transfer artifact, and the Workflow Runs contract table. The single strongest idea is the run-journal contract: "the journal caches completed agent calls only — resume replays them labelled cached, in-flight agents re-run whole; a retried phase restarts its roll-up and failed spend stays on the run meter" — a precise, implementable cost-and-recovery semantics for agent runs that almost no design writing articulates, and it's fully realized in `workflow-run-monitor.tsx`. But every claim in the area is currently ASSERTED; the material is defensible as an original design spec only if it stops implying these are observed conventions and grounds the 6–8 checkable claims above.

- GitHub Copilot: the **agents panel** (available on every github.com page) is described by GitHub as "your mission control center for agentic workflows" — hand off tasks and track existing sessions; the **agents page** lists all sessions; JetBrains got a **unified sessions view** where "each session shows its title, agent type, elapsed time, and status" with filtering by agent type/status. - Cursor 3 (April 2, 2026): the Agents Window lists every active agent session "local or cloud, across all your repos" with the task that started it, target repo, and environment; per-row live info is "current file and last action." Sources: https://cursor.com/changelog/3-0 , https://www.datacamp.com/blog/cursor-3 , https://dev.to/thegdsks/cursor-3-ships-parallel-ai-agents-here-is-the-multi-agent-workflow-that-actually-works-2bk8 (evidence July 2026).
- Claude Code: the lead's terminal shows an **agent panel** below the prompt listing teammates (select, view transcript, message, interrupt, stop); plus a full-screen "agent view" dashboard for background sessions ("what's working, what needs input, what's done"). Source: https://code.claude.com/docs/en/agent-teams (fetched July 2026, describes v2.1.178+); https://www.cloudzero.com/blog/claude-code-agents/ (May 18, 2026).
- OpenAI Codex app: agents run in threads organized by project; changelog lists "improved status indicators for running threads, queued prompts, side chats, and subagents." Sources: https://openai.com/index/introducing-the-codex-app/ , https://developers.openai.com/codex/changelog (Windows release March 4, 2026).
- Verdict: name + task + live status = CONFIRMED convention. The *specific card fields* (idle reason, inline retry, $ column) are not — see CONTRADICTED.

**2. Per-agent-session compute accounting exists — but in tokens/ACUs/credits, never live dollars (partially supports, mostly reframes, the fleet-table Cost column).**
- Devin: Session Insights modal shows **ACU usage** as a key metric per session (post-hoc, not a live column); per-session ACU caps can be set. Source: https://docs.devin.ai/product-guides/session-insights (July 2026).
- GitHub Copilot: opening a session shows "the agent's progress, **token usage**, and session length." Source: https://docs.github.com/en/copilot/how-tos/copilot-on-github/use-copilot-agents/manage-and-track-agents
- Vercel eve (framework runtime lever, primary source, local install): every subagent delegation "gets its own child session and stream. The telemetry matches direct delegation, so existing dashboards and **cost attribution** keep working" — per-child cost attribution is a framework-level contract. Source: `/Users/arielconti/workspace/eve-chat/node_modules/eve/docs/guides/dynamic-workflows.md` and `/Users/arielconti/workspace/eve-chat/node_modules/eve/docs/subagents.mdx` (parent stream carries only `subagent.called` / `subagent.completed`; child progress lives at `GET /eve/v1/session/:childSessionId/stream`).

**3. "Journal caches completed agent calls only; resume replays them; in-flight work re-runs" — this is exactly durable-execution semantics and can be upgraded to CITED cheaply.**
- Inngest: "each step runs once, its result is persisted, and subsequent executions skip completed steps by injecting their stored results." Sources: https://www.inngest.com/docs/learn/how-functions-are-executed , https://www.inngest.com/docs/learn/inngest-steps (July 2026).
- LangGraph time travel: "nodes before the checkpoint are not re-executed (results are already saved)… nodes after the checkpoint re-execute, including any LLM calls." Source: https://docs.langchain.com/oss/python/langgraph/use-time-travel
- Temporal: replay checks generated commands against existing Event History; the UI animates pending events with dashed lines (dashed red = retrying, dashed purple = pending, green = complete) and shows a pending activity's attempt count — also grounds the guide's phase-status glyph vocabulary. Sources: https://docs.temporal.io/workflow-execution/event , https://temporal.io/changelog/updated-event-history-timeline-view-is-now-available
- Vercel Workflow: "every step, input, output, sleep, and error inside a workflow is recorded automatically"; runs inspectable in dashboard Observability → Workflows. Source: https://workflow-sdk.dev/docs/observability , https://vercel.com/docs/workflows
- Eve dynamic workflows (primary): "the whole orchestration counts as one step… if a run parks on a long-running or human-gated child, it resumes where it left off after a restart" (same file as above).
- Caveat worth one sentence in the guide: no durable-execution UI literally labels replayed steps "cached" — that word is the guide's invention; the semantics are standard.

**4. Retry-only-failed / skip-preserves-record is standard CI semantics.**
- GitHub Actions: "Re-run failed jobs" re-runs only failed jobs (web UI dropdown + `gh run rerun --failed`); re-runs allowed up to 30 days, max 50 per run; the prior failed attempt remains viewable as a previous run attempt. Sources: https://docs.github.com/en/actions/how-tos/manage-workflow-runs/re-run-workflows-and-jobs , https://github.blog/changelog/2022-03-17-github-actions-re-run-only-failed-or-individual-jobs/

**5. Accept/reject lifecycles for inter-agent work DO exist — at two specific layers the guide could cite.**
- Protocol layer — Google A2A: task lifecycle `submitted → working → input-required/auth-required → completed / failed / canceled / rejected`; `rejected` = "the task was rejected by the agent and was not started." Source: https://a2a-protocol.org/latest/specification/ (v0.3.0+).
- Product layer — Claude Code agent teams **plan approval**: "the teammate works in read-only plan mode until the lead approves… The lead reviews the plan and either approves it or rejects it with feedback. If rejected, the teammate stays in plan mode, revises… and resubmits." Also teammate shutdown requests: "The teammate can approve, exiting gracefully, or reject with an explanation." Source: https://code.claude.com/docs/en/agent-teams — this is the closest shipping analog to the HandoffPacket's negotiated lifecycle (though it gates *plans in*, not *outputs out*).
- Human-in-the-loop layer — LangChain **Agent Inbox**: interrupts render as items with accept / edit / respond / ignore actions, config-gated per interrupt (`allow_accept`, `allow_ignore`…). Source: https://github.com/langchain-ai/agent-inbox

**6. "OpenAI-style handoff = silent one-way transfer" (the naming-collision staleness risk is real).**
- OpenAI Agents SDK: handoffs are represented to the LLM as `transfer_to_<agent_name>` tools; "a one way transfer… we immediately start execution on that new agent" — no acceptance step, no negotiation. Sources: https://openai.github.io/openai-agents-python/handoffs/ , https://developers.openai.com/api/docs/guides/agents/orchestration

**7. Inter-agent messages render as attributed transcripts/logs, not as the guide's from→to chat thread — with one dev-tool exception.**
- Claude Code agent teams: communication is a **mailbox** (`SendMessage`); you read a teammate's own session transcript (Enter on the agent panel row / split pane); receiving agents are told a message "came from another Claude session, not from you." Source: https://code.claude.com/docs/en/agent-teams
- AutoGen Studio (the exception, a low-code dev tool): "chat with the workflow, observe all agent messages," review the "inner monologue" plus per-run cost/turn/token profiling. Sources: https://www.microsoft.com/en-us/research/blog/introducing-autogen-studio-a-low-code-interface-for-building-multi-agent-workflows/ , https://microsoft.github.io/autogen/0.2/blog/2023/12/01/AutoGenStudio/
- Copilot session logs "show Copilot's internal reasoning and the tools it used" — trace-style. Source: https://docs.github.com/en/copilot/how-tos/copilot-on-github/use-copilot-agents/manage-and-track-agents
- Verdict: the guide's Shared Context mode (typed artifact feed with attribution) is the realistic default; Direct chat mode exists only in debugging tools.

**8. Agent status-state vocabularies are a real published convention — Linear AIG.**
- Linear agent sessions have 6 user-visible states: `pending, active, error, awaitingInput, complete, stale`; agents "should clearly indicate whether they're thinking, waiting for input, executing, or finished"; must acknowledge invocation immediately; identity "must never be mistaken for a person." Sources: https://linear.app/developers/aig , https://linear.app/changelog/2025-07-30-agent-interaction-guidelines-and-sdk (July 30, 2025). This is the strongest citable grounding for the Agent Cards three-state model — and note `awaitingInput` and `stale` are richer than the guide's idle/error split.
